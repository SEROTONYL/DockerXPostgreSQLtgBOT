# ARCHITECTURE

## Цель документа

- Это краткий набор архитектурных правил и границ слоёв.
- Документ нужен, чтобы проект не превратился в кашу при росте кода и фич.
- По этому документу ревьюер проверяет PR на архитектурные нарушения.

## Карта слоёв (Layer map)

- `cmd/*` — entrypoints приложения; только запуск, чтение конфигурации и вызов сборки приложения.
- `internal/app` — composition root; создание реализаций, wiring зависимостей, запуск сценариев.
- `internal/bot` — ядро обработки апдейтов: парсинг, фильтры, диспатч команд; без бизнес-логики фич.
- `internal/commands` — командный роутер и контекст хендлеров; контрактная точка между ядром и фичами.
- `internal/features/*` — бизнес-логика и UI-флоу конкретных фич; orchestration действий пользователя.
- `internal/telegram` — единый шлюз к Telegram API: send/edit/ack, нормализация и классификация ошибок.
- `internal/repo|storage|db` (если есть) — доступ к БД/хранилищам; без Telegram-типов и без UI-логики.

## Правила зависимостей (Dependency rules)

- Разрешённая верхнеуровневая цепочка: `cmd/* -> internal/app -> остальные пакеты через app`.
- `internal/app` может импортить любые внутренние пакеты, потому что он выполняет роль склейки.
- `internal/bot` может импортить:
  - `internal/commands`
  - `internal/telegram`
  - `internal/bot/filters`
- `internal/bot` НЕ может импортить `internal/features/*`.
- `internal/features/*` может импортить:
  - `internal/commands` (контекст, интерфейсы, типы)
  - `internal/telegram` (Ops-интерфейс/абстракции шлюза)
  - репозитории/хранилища (`internal/repo|storage|db`)
- `internal/features/*` НЕ может импортить:
  - `cmd/*`
  - `internal/app`
- Репозитории (`internal/repo|storage|db`) НЕ должны импортить:
  - `tgbotapi`
  - `internal/telegram`
  - `internal/bot`

## Запреты (Hard bans)

- Нельзя напрямую вызывать Telegram API из фич и репозиториев.
- Любой Telegram-вызов должен идти только через `internal/telegram.Ops`.
- Нельзя чтобы `internal/bot` зависел от `internal/features/*`.
- Нельзя тащить типы `tgbotapi` ниже слоя `internal/telegram`.
- Репозитории и доменные структуры не должны содержать Telegram-типы.
- `internal/app` — единственное место, где создаются concrete-реализации и связываются зависимости.

## Инварианты UI/Callback

- Любой callback handler обязан в начале делать `AnswerCallback` (ACK).
- Рендер экранов выполняется через единый helper в стиле `edit-or-send` (редактировать существующее сообщение, иначе отправить новое).
- После успешного действия не создавать «второе сообщение» без явной необходимости.
- Ошибки UI обрабатывать предсказуемо: ACK остаётся обязательным даже при ошибке бизнес-операции.

## PR checklist по архитектуре

- Проверены импорты: не нарушены зависимостные границы между слоями.
- В `cmd/*` нет бизнес-логики и Telegram-вызовов.
- Новые фичи подключаются через router/контракты `internal/commands`.
- `internal/bot` не импортит `internal/features/*`.
- Telegram-вызовы в фичах идут только через `internal/telegram.Ops`.
- Репозитории не знают про Telegram (`tgbotapi`, `internal/telegram`, `internal/bot` отсутствуют).
- Callback-обработчики делают ACK в начале.
- UI-рендер не плодит лишние сообщения, используется стратегия `edit-or-send`.
